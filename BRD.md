## 一、项目背景与愿景

### 1.1 为什么要做这个项目

在日常工作和生活中，越来越需要一个 **懂我、记我、陪我的 AI 伙伴**——不只是冷冰冰的工具型助手，而是一个能够：

- 记住我告诉过它的事情，哪怕隔了几天几周
- 感知我的情绪状态，用合适的方式回应我
- 拥有自己的"性格"，像一个真实的朋友
- 帮我查天气、查路线、搜周边，像一个能干的生活助理

市面上虽然有不少 AI 对话产品，但它们大多是通用型助手或者封闭的商业产品。作为技术人员，**自己造一个完全可控、可持续演进、可深度定制的 AI 助手**既有技术乐趣，也能真正解决自己的需求。

### 1.2 项目愿景

> 打造一个**有记忆、有情感、有能力**的个人 AI 助手，从简单的"陪聊"逐步进化成"懂我的智能生活伙伴"。

### 1.3 核心设计原则

| 原则 | 说明 |
|------|------|
| **本地可控** | 数据在本地，隐私由自己掌控 |
| **渐进演进** | 从 MVP 开始，逐步叠加能力，不过度设计 |
| **模块解耦** | 每个功能模块可独立启用/禁用 |
| **成本可控** | 优先选择性价比高的 API，避免跑飞账单 |
| **自用优先** | 先满足自己，再考虑通用化 |

### 2.1 目标用户

## 四、Phase 1：核心对话引擎（已完成 ✅）

> **目标**：搭建最基本、可用的 AI 助手，能聊天、能记住我、能感知情绪。

### 4.1 ✅ BR-001 智能对话能力

| 属性 | 内容 |
|------|------|
| **需求描述** | 用户输入文字消息，AI 助手基于预设人设生成自然、贴心的文字回复 |
| **优先级** | P0 |
| **验收标准** | ① 支持多轮对话，能理解上下文；② 回复符合设定的人设（性格、说话风格）；③ 回复简洁自然，不啰嗦 |
| **技术实现** | DeepSeek API（`deepseek-v4-flash`）通过 LangChain `ChatOpenAI` 封装 |
| **对应代码** | `src/agent/llm.py`、`src/agent/companion.py` |

### 4.2 ✅ BR-002 可自定义的角色人设

| 属性 | 内容 |
|------|------|
| **需求描述** | 用户可以自定义AI助手的名字、性格特点、背景故事 |
| **优先级** | P0 |
| **验收标准** | ① 支持配置名字（如"小梦"）；② 支持配置性格（如"温柔、善解人意、有点俏皮"）；③ 支持配置背景故事；④ 配置通过环境变量/.env 文件读取 |
| **技术实现** | Pydantic Settings 管理 `companion_name`、`companion_personality`、`companion_backstory`，注入 System Prompt |
| **对应代码** | `src/config/settings.py`、`src/agent/companion.py:_build_system_prompt()` |

### 4.3 ✅ BR-003 短期记忆（对话上下文）

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手能在同一会话中记住最近说过的话，理解对话上下文 |
| **优先级** | P0 |
| **验收标准** | ① 支持 N 轮对话上下文窗口（可配置，默认 20 条消息）；② 超出窗口时自动裁剪最早的消息；③ 能引用上文中用户提到的信息 |
| **技术实现** | LangChain `MessagesPlaceholder` + 滑动窗口裁剪 |
| **对应代码** | `src/agent/companion.py:_build_prompt()`、短期记忆裁剪逻辑 |

### 4.4 ✅ BR-004 长期记忆（跨会话记忆）

| 属性 | 内容 |
|------|------|
| **需求描述** | 即使关闭服务重新打开，AI助手仍然记得之前告诉过它的重要信息 |
| **优先级** | P1 |
| **验收标准** | ① 用户关闭服务后重新打开，AI助手能回忆起之前对话中提到的重要事实；② 基于语义检索，不依赖关键词匹配；③ 每次检索最多召回 k 条最相关记忆（可配置，默认 3） |
| **技术实现** | ChromaDB 向量数据库 + DashScope `text-embedding-v3` 向量化，语义检索 |
| **对应代码** | `src/agent/memory.py` |

### 4.5 ✅ BR-005 情感感知与自适应回复

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手能识别用户当前的情绪，并用匹配的语气回复 |
| **优先级** | P1 |
| **验收标准** | ① 支持识别 5 种情绪：开心、难过、愤怒、焦虑、平静；② 根据情绪自动调整回复语气（如难过时温柔安慰，开心时活泼俏皮）；③ 情绪标签在前端 UI 可见 |
| **技术实现** | 额外 LLM 调用做情绪分析 → 注入情绪调整指令到 System Prompt |
| **对应代码** | `src/agent/emotion.py` |

### 4.6 ✅ BR-006 流式输出（打字机效果）

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手的回复像真人打字一样逐字显示，而不是一次性全部弹出 |
| **优先级** | P1 |
| **验收标准** | ① 前端逐字渲染；② 流式推送采用 SSE 协议；③ 发送期间显示打字指示器动画 |
| **技术实现** | FastAPI SSE (`text/event-stream`) + 前端 `ReadableStream` |
| **对应代码** | `src/ui/server.py`、`src/ui/static/index.html` |

### 4.7 ✅ BR-007 对话历史持久化

| 属性 | 内容 |
|------|------|
| **需求描述** | 关闭服务再打开时，上次的对话记录还在界面上 |
| **优先级** | P0 |
| **验收标准** | ① 服务重启后自动恢复上次对话记录；② 支持清除对话历史；③ 数据格式可读 |
| **技术实现** | JSON 文件存储（`./data/chat_history.json`） |
| **对应代码** | `src/agent/companion.py:_save_history()`、`_load_history()` |

### 4.8 ✅ BR-008 Web 图形界面

| 属性 | 内容 |
|------|------|
| **需求描述** | 提供美观、现代的 Web 聊天界面 |
| **优先级** | P0 |
| **验收标准** | ① 支持发送消息、显示回复；② 美观的玻璃拟态（Glassmorphism）视觉风格；③ 动态背景动画；④ 响应式布局；⑤ 显示AI助手头像和在线状态；⑥ 显示当前情绪标签 |
| **技术实现** | 原生 HTML/CSS/JS（零框架依赖）、CSS 动画、渐变背景 |
| **对应代码** | `src/ui/static/index.html` |

### 4.9 ✅ BR-009 对话重置与数据清除

| 属性 | 内容 |
|------|------|
| **需求描述** | 用户可以重置当前对话（清空上下文），或清除所有历史数据 |
| **优先级** | P1 |
| **验收标准** | ① "重置对话"清空短期记忆，不删除持久化文件和长期记忆；② "清除所有数据"删除对话历史 + ChromaDB 全部长期记忆，且需二次确认 |
| **技术实现** | REST API `/api/reset` 和 `/api/clear` |
| **对应代码** | `src/ui/server.py`、`src/agent/companion.py:reset()`、`clear_all_data()` |

### 4.10 ✅ BR-010 MCP 工具集成（持续扩展中）

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手能通过 MCP 协议调用外部工具，逐步扩展自身能力。已集成：高德地图（12 个工具：地理编码、逆地理编码、天气查询、周边搜索、路线规划等）+ Tavily 联网搜索（实时资讯、网页搜索） |
| **优先级** | P2 |
| **验收标准** | ① ✅ MCP 协议接入高德地图 + Tavily；② ✅ LLM 自动选择正确工具并执行；③ ✅ 事件循环死锁已修复；④ ✅ 单个 MCP 连接失败自动跳过，互不拖累；⑤ ✅ 工具调用失败不崩溃，降级处理；⑥ ✅ `.env` 配置 API Key 即自动启用；⑦ 支持 JSON 配置文件扩展更多 MCP |
| **技术实现** | MCP SDK（`mcp` 包）+ 自定义 `_MCPTool` 封装为 LangChain Tool + Agent 工具调用循环（最多 5 轮迭代） |
| **对应代码** | `src/agent/tools.py`、`src/agent/companion.py:_run_with_tools()` |

## 五、Phase 2：交互体验升级（部分完成 🚧）

> **目标**：让聊天更丰富、更自然、更像"真人"，提升日常使用体验。

### 5.1 ✅ BR-011 Markdown 渲染

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手回复中的 Markdown 格式文本（列表、粗体、链接等）正确渲染显示 |
| **优先级** | P1 |
| **验收标准** | ① 支持粗体、斜体、列表、代码块、链接等常见格式；② 代码块支持语法高亮 |
| **技术方向** | `marked.js` + `highlight.js` CDN 引入前端，自定义 `marked.Renderer` 集成语法高亮，流式期间 textContent 避免闪烁，完成后 Markdown 渲染 |

### 5.2 ✅ BR-012 位置感知与智能场景触发

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手获取用户 GPS 位置，用户说"今天天气怎么样"即自动反查城市 + 调用天气工具，无需手动告知所在地 |
| **优先级** | P1 |
| **验收标准** | ① 首次打开页面弹出位置授权；② 授权后每次消息自动附带经纬度；③ 高德逆地理编码反查城市名，注入 System Prompt；④ 天气等场景对话 LLM 自动识别位置并调用工具；⑤ 出差到新城刷新后位置自动更新；⑥ 授权拒绝/GPS 不可用时优雅降级 |
| **典型对话** | "今天天气怎么样？" → 自动获取 "杭州市" → 调用天气工具 → "杭州多云 15-22°C 适合出门～" |
| **技术方向** | 前端 `navigator.geolocation` 获取 GPS → 每次消息附带 `{lat, lng}`；后端 `_reverse_geocode()` 直接调高德 REST API（不走 MCP，避免异步复杂度）反查城市名 → 注入 System Prompt `【位置感知】` → LLM 自动识别天气/周边等场景并调用工具 |

### 5.7 🚧 BR-017 文档上传与阅读

| 属性 | 内容 |
|------|------|
| **需求描述** | 用户可以将 PDF、Word、TXT、Markdown 等文档拖拽/上传到聊天中，AI助手读取文档内容后能基于文档内容回答问题、总结摘要、提取关键信息 |

## 六、Phase 3：深度记忆与理解（规划中 🚧）

> **目标**：让AI助手真正"懂我"，从记住事实到理解偏好、构建用户画像。

### 6.1 🚧 BR-018 用户画像自动构建

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手从对话中自动提取并沉淀用户信息（名字、年龄、所在地、兴趣爱好、工作等），形成动态更新的用户画像 |
| **优先级** | P1 |
| **验收标准** | ① 自动识别用户在对话中透露的个人信息；② 画像结构化存储（JSON 或独立 ChromaDB collection）；③ 画像条目支持增删改查；④ 每次对话中自动参考画像信息 |
| **技术方向** | LLM 结构化输出提取用户信息 + ChromaDB 独立 collection |

### 6.2 🚧 BR-019 记忆时间衰减

| 属性 | 内容 |
|------|------|
| **需求描述** | 越久远的记忆召回优先级越低，越近期的记忆越容易被检索到 |
| **优先级** | P2 |
| **验收标准** | ① 记忆存储时记录时间戳；② 检索时根据时间差计算衰减权重；③ 支持配置衰减曲线 |
| **技术方向** | ChromaDB metadata 增加时间戳 + 检索后加权排序 |

### 6.3 🚧 BR-020 记忆重要性评分

| 属性 | 内容 |
|------|------|
| **需求描述** | 不是所有对话都值得长期记住——AI助手应该自动判断哪些信息重要 |
| **优先级** | P2 |
| **验收标准** | ① 自动对每轮对话内容做重要性评分（0-1）；② 高重要性记忆优先被召回；③ 低重要性记忆可被淘汰 |
| **技术方向** | LLM 评分 + metadata 存储 + 检索加权 |

### 6.4 🚧 BR-021 对话话题聚类

| 属性 | 内容 |
|------|------|
| **需求描述** | 将历史对话按话题自动归类（如"工作"、"情感"、"生活"、"技术"），方便回顾 |
| **优先级** | P2 |
| **验收标准** | ① 自动将对话片段归入话题标签；② 话题视图可浏览该分类下的对话；③ 支持手动修正标签 |
| **技术方向** | 文本聚类（K-Means/HDBSCAN）+ LLM 标签生成 |

### 6.5 🚧 BR-022 每日/每周摘要生成

| 属性 | 内容 |
|------|------|
| **需求描述** | AI助手每天或每周自动生成一份对话摘要，回顾这段时间聊了什么、用户情绪变化趋势 |
| **优先级** | P2 |
| **验收标准** | ① 每日摘要：总结今日聊天要点和情绪曲线；② 每周摘要：汇总本周话题、心情变化、重要事件；③ 可选择展示在 UI 中 |
| **技术方向** | 定时任务 + LLM 摘要生成 |

## 十一、持续演进与风险管控

> MCP 工具集成是项目长期演进的核心增长点。随着工具数量增加，需要提前规划治理策略，避免"工具越多越乱"。

### 11.1 MCP 工具清单（当前已集成）

| 服务 | 工具数量 | 典型能力 | API 依赖 |
|------|----------|----------|----------|
| 高德地图 AMap | 12 个 | 地理编码、逆地理编码、天气、周边搜索、路线规划、IP 定位等 | `AMAP_MAPS_API_KEY` |
| Tavily 搜索 | 1 个 | 实时联网搜索、新闻检索、网页摘要 | `TAVILY_API_KEY` |

### 11.2 规模增长的风险矩阵

| 风险 | 当前状态（14 工具） | 30+ 工具时 | 50+ 工具时 |
|------|---------------------|------------|------------|
| **工具误调用** | 🟢 低风险：命名前缀清晰（`maps_*` vs `tavily_*`） | 🟡 中风险：可能出现跨域误判 | 🔴 高风险：需路由分流 |
| **上下文膨胀** | 🟢 System Prompt 工具列表约 1K tokens | 🟡 工具描述挤占上下文，压缩记忆空间 | 🔴 需动态裁剪 |
| **多工具串行延迟** | 🟢 通常 1 轮工具调用 | 🟡 复杂查询可能触发 2-3 轮 | 🔴 延迟达到用户可感知阈值 |
| **API 费用叠加** | 🟢 已实现用量监控 + 每日预算上限（BR-032） | 🟡 频繁多轮工具调用增加 token 消耗 | 🔴 需更细粒度的工具调用成本控制 |
| **工具冲突/死锁** | 🟢 已修复跨事件循环死锁问题 | 🟡 npx 子进程管理复杂度增加 | 🔴 需进程健康监控 |

### 11.3 治理策略（按优先级排序）

#### 策略 1：System Prompt 路由提示（P1，立即可做）

当工具超过 ~15 个时，在 System Prompt 中注入路由规则，减少 LLM 的工具选择成本：

```
【工具选择提示】
• 天气、地图、导航、周边搜索 → 使用 maps_* 系列工具
• 实时资讯、新闻、百科知识搜索 → 使用 tavily_* 系列工具
• 不确定时应优先用最匹配的工具，避免同时调用同类工具
```

#### 策略 2：动态工具裁剪（P2，30+ 工具时启用）

在 `_prepare_context` 中根据用户意图预筛选工具列表。不在本轮需求域的工具不注入 System Prompt，既省 token 又减少误判：

| 用户意图 | 注入的工具 |
|----------|-----------|
| 天气/位置/路线 | `maps_weather`、`maps_geo`、`maps_direction` |
| 新闻/知识查询 | `tavily_search` |
| 日常闲聊 | 不注入任何工具 |

#### 策略 3：工具分组与优先级（P2）

为每个工具标注领域标签和调用成本，LLM 可根据标签快速路由：

```python
# 未来 _MCPTool 可扩展的元数据
tool.domain = "map" | "search" | "utility"
tool.cost_level = "low" | "medium"  # 影响 LLM 是否主动调用
```

#### 策略 4：工具健康监控（P2，npx 子进程管理）

监控 npx 子进程的存活状态，挂掉后自动重启，避免用户感知到工具不可用：

- 进程心跳检测（每隔 N 秒 ping）
- 异常自动重启
- 降级标记（临时移除挂掉的工具，恢复后自动加回）

### 11.4 记忆系统的演进风险

| 风险 | 说明 | 治理措施 |
|------|------|----------|
| 向量膨胀 | ChromaDB 存储量随对话增长无上限 | 内存遗忘策略（BR-019）自动淘汰低价值记忆 |
| 检索质量衰减 | 海量数据中噪声增加，召回精度下降 | 重要性评分（BR-020）加权排序 |
| 存储成本 | 单文件体积增长 | 定期压缩 / 分片存储 |

### 11.5 成本增长的长期管控

| 阶段 | 工具数量 | 预估每次对话额外成本 | 管控措施 |
|------|----------|---------------------|----------|
| 当前 | 14 个 | 0-2 次工具 LLM 调用 | 无需额外管控 |
| 中期 | 15-30 个 | 1-3 次工具 LLM 调用 | 动态裁剪 + 路由提示 |
| 远期 | 30+ 个 | 2-5 次工具 LLM 调用 | 预算上限 + 工具调用次数限制 |

> 底线原则：MCP 工具是增强不是必须。任何一个工具挂掉都不应影响核心对话能力——这是从 Day 1 就贯彻的设计准则。

### 11.6 Prompt 持续调优（持续性任务）

每次新增能力或工具后，需持续检查AI助手的**输出质量**是否偏离预期，重点观察四个维度：

| 维度 | 关注点 | 典型翻车案例 |
|------|--------|-------------|
| **人设一致性** | 回复是否保持角色风格 | 调完天气工具后变成 Siri 口吻 |
| **回复长度** | 新闻/搜索类任务是否过长 | Tavily 搜 5 条新闻写了 400 字小作文 |
| **关怀自然度** | 情感关怀是否生硬插入 |"今天科技巨震……别担心工作哦"（前后无关） |
| **工具融合度** | 工具数据是否自然融入口吻 | 直接甩JSON："今天 36°C，湿度 45%，风速 3m/s……" |

**调优节奏**：

- 每次新增 MCP 工具后，跑 3-5 轮对话回归验证，确认人设没跑偏
- 工具链路变复杂时（多工具协作），重点检查回复长度和融合度
- 每积累约 5 个调整点后，重新审查 System Prompt 是否需要结构性优化（而非不断追加补丁）
- 调优方式：修改 `src/agent/companion.py:_build_system_prompt()` 或 `src/agent/emotion.py` 情绪调整指令

       ▼

│                  │  ← JSON 文件持久化

## 附录 A：术语表

| 术语 | 说明 |
|------|------|
| **短期记忆** | 当前会话中的最近 N 轮对话上下文，服务重启即丢失，负责维持对话连贯性 |
| **长期记忆** | 存储在 ChromaDB 向量数据库中的历史对话，跨会话持久化，负责"记住用户" |
| **情绪感知** | 用 LLM 分析用户消息的情绪（5 类），并动态调整回复策略 |
| **MCP** | Model Context Protocol，一种标准化的 AI 工具调用协议，用于连接外部工具服务器 |
| **SSE** | Server-Sent Events，服务器向客户端单向推送数据的协议，用于流式输出 |
| **ChromaDB** | 开源向量数据库，用于语义检索 |
| **Embedding** | 将文本转换为向量（数字数组），向量距离=语义相似度 |
| **滑动窗口** | 短期记忆的管理策略：保留最新 N 条，超出则丢弃最早的消息 |
