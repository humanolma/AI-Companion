# 面试准备：项目搭建难点与解决方案

> 适用于：面试官问"搭建项目过程中遇到了哪些难点 / 你是怎么解决的"

---

## 难点一：Gradio 版本兼容性问题（UI 层）

### 问题描述
项目使用 Gradio 6.x 作为 Web UI，但 Gradio 不同大版本之间的 API 变化较大：
- `Chatbot` 组件的 `type` 参数在 6.x 中被移除
- `show_copy_button` 参数在部分 6.x 版本中不存在
- `css` 参数从 `gr.Blocks()` 移到了 `launch()` 方法
- `Chatbot` 的数据格式在 6.x 中默认要求 `messages` 格式（dict 列表），而不是旧版的 `tuples` 格式

导致启动时报 `TypeError: __init__() got an unexpected keyword argument` 和 `Data incompatible with messages format` 错误。

### 解决方案
1. **查阅 Gradio 官方文档**，确认当前安装版本（6.19.0）的 API 签名
2. **移除不支持的参数**（`type`、`show_copy_button`）
3. **将 `css` 参数移到 `launch()`**
4. **统一 Chatbot 数据格式**为 `messages` 格式（list of dict with `role` and `content` keys）

### 技术收获
- 第三方库版本管理的重要性（应在 `requirements.txt` 中固定版本号）
- 阅读官方文档和源码 traceback 来定位 API 变化
- 前后端数据格式约定需要前后一致

## 难点二：LangChain + DeepSeek API 适配

### 问题描述
DeepSeek API 兼容 OpenAI 格式，但 `base_url` 需要指向 DeepSeek 的服务器 `https://api.deepseek.com/v1`。
如果 `base_url` 配置错误，会导致连接超时或返回模型不存在的错误。

### 解决方案
1. 使用 `langchain-openai` 的 `ChatOpenAI` 类，通过 `base_url` 参数指向 DeepSeek
2. 在 `.env` 中统一管理 API Key 和 base_url，通过 `pydantic-settings` 自动加载
3. 封装 `get_llm()` 工厂函数，方便后续切换模型

### 技术收获
- 使用 `pydantic-settings` 管理配置，比原生 `os.getenv()` 更优雅，支持类型校验
- LangChain 的 LLM 抽象层设计很好，切换模型只需改一处

## 难点三：对话记忆管理（短期 vs 长期）

### 问题描述
虚拟伴侣需要记住对话历史，但 LangChain 的 `MessagesPlaceholder` 会把所有历史塞进 context window，存在两个问题：
1. **Token 消耗**：历史越长，每次调用费用越高
2. **上下文截断**：超过模型 token 限制后会报错

### 当前方案（短期记忆）
- 在 `CompanionAgent` 内部维护 `self.history` 列表（存储 `HumanMessage` / `AIMessage`）
- 每次对话后追加，并限制最大长度（最近 20 条）
- 会话重置时清空

### 已实现的长期记忆方案
- 使用 **ChromaDB** 向量数据库存储历史对话
- 每次用户输入时，先向量检索最相关的历史片段（top-k=3），再拼入 system prompt
- Embedding 使用本地 `sentence-transformers`（all-MiniLM-L6-v2），无需外部 API
- 自实现 `LocalEmbeddings` 包装类，继承 `langchain_core.embeddings.Embeddings` 接口

### 遇到的问题
1. **DeepSeek 不支持 embeddings API** → 改用本地 sentence-transformers 模型
2. **langchain 1.x 移除了 `SentenceTransformerEmbeddings`** → 自实现包装类，更稳定
3. **缺少 `langchain-chroma` 包** → 安装并加入 requirements.txt

### 测试验证
- 第一轮对话：用户介绍自己（名字、年龄、城市、爱好）
- 重置短期记忆（模拟第二天对话）
- 第二轮对话：用户问"你还记得我叫什么吗"
- 结果：Agent 通过 ChromaDB 检索成功回忆出用户信息 ✅

### 技术收获
- 对 LLM 应用中的"记忆"问题有了系统理解：短期（上下文）vs 长期（外部存储）
- 向量检索是 RAG 的核心，本项目实现了完整的记忆检索流程
- 学会了 LangChain 的 Embeddings 抽象接口设计

## 难点四：项目结构设计（工程化）

### 问题描述
一开始把所有代码写在一个文件里，导致：
- 配置、Prompt、Agent 逻辑混在一起，难以维护
- 无法单独测试某个模块
- 不符合开源项目的代码规范

### 解决方案
重构为分层结构：
```
src/
├── config/   # 配置（环境变量、模型参数）
├── agent/    # Agent 核心逻辑（LLM 初始化、对话管理）
├── prompts/  # Prompt 模板（人设、系统提示词）
└── ui/       # Web 界面（Gradio）
```

每个模块：
- 有独立的 `__init__.py`
- 关键类/函数有 docstring
- 通过 `from src.xxx import YYY` 进行模块间引用

### 技术收获
- 学会了 Python 的模块导入机制（`__init__.py` 的作用）
- 项目结构清晰后，后续添加功能（如长期记忆、情感感知）可以更聚焦

## 难点五：情感感知模块设计

### 问题描述
虚拟伴侣需要识别用户情绪并调整回复语气，但面临两个设计决策：
1. **情绪检测方式**：用关键词匹配（快但不准确）还是用 LLM 分析（准确但多一次 API 调用）？
2. **情绪如何影响回复**：直接改 prompt 还是用后处理？

### 解决方案
1. **用 LLM 做情绪分析**：定义 5 种情绪（happy/sad/angry/anxious/neutral），用一个简短的 system prompt 让 LLM 只返回标签单词
2. **动态调整 system prompt**：根据检测到的情绪，在 system prompt 中追加回复风格指导（如"用户难过时先表达关心，不要急着给建议"）
3. **缓存机制**：对相同输入缓存检测结果，避免重复调用

### 设计亮点
- 情绪检测和回复生成分离，职责清晰
- 降级策略：LLM 调用失败时返回 neutral，不影响主流程
- Web UI 实时显示检测到的情绪，用户能看到 Agent 的"感知"过程

### 测试结果
5 种情绪（开心/难过/愤怒/焦虑/平静）全部检测正确，回复风格也相应调整。

### 技术收获
- 理解了 LLM 应用的"感知-决策-执行"模式
- 情绪检测本质是一个小型的 LLM 分类任务
- 动态 prompt 调整比后处理更自然，因为 LLM 能在生成时就考虑情绪因素

## 难点六：流式输出实现

### 问题描述
一次性返回完整回复时，用户需要等待几秒才能看到结果，体验不佳。需要实现"打字机效果"，让回复逐字显示。

### 解决方案
1. **Agent 层**：用 LangChain 的 `chain.stream()` 替代 `chain.invoke()`，返回一个 generator，逐 chunk yield 文本片段
2. **UI 层**：Gradio 的事件处理函数改为 generator 函数，每次 yield 更新 Chatbot 的最后一条消息
3. **记忆更新**：流式结束后，把完整回复一次性存入短期/长期记忆

### 关键代码
```python
# Agent 层
def chat_stream(self, user_input):
    full_response = ""
    for chunk in self.chain.stream({...}):
        full_response += chunk.content
        yield chunk.content
    # 流结束后更新记忆
    self.history.append(...)

# UI 层
def chat_stream(user_input, history):
    history.append({"role": "user", "content": user_input})
    history.append({"role": "assistant", "content": ""})
    for piece in agent.chat_stream(user_input):
        history[-1]["content"] += piece
        yield history, "", gr.Markdown()
```

### 技术收获
- 理解了 LLM 流式输出的原理：SSE（Server-Sent Events），服务器逐步推送 token
- Gradio 的 generator 函数天然支持流式更新 UI
- 流式输出不只是体验优化，还能降低首字延迟（TTFT），提升感知性能

## 面试官可能追问的问题 + 准备答案

### Q1：为什么选 LangChain 而不是原生 OpenAI SDK？
**A**：LangChain 提供了 Agent、记忆、工具调用等抽象，后续扩展功能（如接入向量数据库、工具调用）会更方便。如果是生产环境且功能简单，原生 SDK 更轻量。

### Q2：DeepSeek 和 OpenAI 比有什么优势？
**A**：DeepSeek 成本低（约 OpenAI 的 1/10），中文能力强，且 API 完全兼容 OpenAI 格式，切换成本低。

### Q3：对话记忆你是怎么设计的？
**A**：短期用 LangChain 的 `MessagesPlaceholder` 维护最近 20 轮对话；长期用 ChromaDB 做向量检索，每次对话前检索 top-3 相关历史拼入 prompt。因为 DeepSeek 不支持 embeddings API，我用本地 sentence-transformers 模型做向量化，自实现了 LangChain 的 Embeddings 接口。测试验证了跨会话的记忆检索功能。

### Q4：如果让你部署到生产环境，你会怎么做？
**A**：
1. 用 FastAPI 封装 Agent，提供 HTTP API
2. 用 Docker 容器化
3. 用 GitHub Actions 做 CI/CD
4. 长期记忆用 Redis + 向量数据库
5. 接入日志和监控（如 LangSmith）
